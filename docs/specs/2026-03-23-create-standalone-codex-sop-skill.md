# 任务规范：创建可独立安装的 Codex SOP Skill

## 背景

当前仓库里已有一个偏“统一入口”的半封装 skill 方案，但它仍依赖本仓库中的多份 SOP 文件和子 skill。新的目标是生成一个真正可独立安装的 skill，使其被复制到其他项目的本地 skills 目录后，就可以通过 `$` 指令直接调用，而不依赖本仓库中的其他 skill 目录。

## 目标

新增一个自包含的 skill 文件夹，支持在其他本地项目中单独安装和直接调用，并覆盖当前 SOP 的核心使用路径。

## 范围

- 新增一个独立 skill 文件夹
- skill 使用单一入口描述 `spec`、`task`、`test`、`facts` 四类核心场景
- skill 内容自包含，不依赖本仓库下其他 skill 文件夹
- skill 结构满足可被 `$skill-name` 直接调用的基本要求

## 非目标

- 不重构现有 `sop-workflow-pack`
- 不删除或替换已有 skill
- 不增加复杂脚本、自动安装器或发布流程
- 不修改无关模板或事实文档
- 不把整个仓库重新组织成插件系统

## 目标文件或区域

- `skills/`
- `docs/specs/`

## 约束

- 变更必须尽可能小
- 新 skill 必须单独可复制、单独可触发、单独可理解
- 默认输出语言保持中文
- 验证优先使用 skill 自身结构校验；只有结构校验不足时才补充内容级检查

## 实现方式

1. 创建一份新 spec 记录该任务边界
2. 使用标准 skill 初始化流程生成新 skill 骨架
3. 将必要 SOP 指导收敛进一个自包含的 `SKILL.md`
4. 确保 skill 的 frontmatter、文件夹命名和 UI 元数据满足直接调用要求

## 验证计划

### 黑盒验证

最小回归敏感测试集应优先覆盖最容易破坏“可独立安装、可直接调用”的用户可见能力：

1. 技能可见性检查
   - 检查 `skills/codex-sop-standalone/` 是否存在，且入口文件为 `SKILL.md`。
   - 预期结果：skill 目录和入口文件都存在，名称与 `$codex-sop-standalone` 一致。
2. 直接调用元数据检查
   - 检查 `skills/codex-sop-standalone/agents/openai.yaml` 是否存在，并声明 `$codex-sop-standalone` 的默认调用提示。
   - 预期结果：skill 具备直接调用所需的 UI 元数据，而不是只能作为文档存在。
3. 自包含能力检查
   - 读取 `SKILL.md`，确认其已经内置 `spec`、`task`、`test`、`facts` 四类场景的最小 SOP 指导。
   - 预期结果：复制该单个 skill 文件夹到其他项目后，仍能理解核心 workflow，而不需要额外复制本仓库其他 skill 才能使用。
4. 语言与边界检查
   - 检查 `SKILL.md` 是否明确默认中文输出，并保留“小范围、显式验证、条件式事实写回”的行为边界。
   - 预期结果：skill 在其他项目中使用时，仍保持与本仓库一致的默认输出与范围控制。

### 白盒验证

默认不增加白盒测试。

仅当黑盒检查无法确认自包含性时，才补充一项最小白盒检查：

1. 硬依赖引用检查
   - 检查 `SKILL.md` 是否把本仓库下其他 skill 文件夹声明为安装时必须存在的前置依赖。
   - 作用：防止“看起来是独立 skill，实际仍然依赖其他 skill 才能工作”的回归。

## 完成标准

- 存在一个新的独立 skill 文件夹
- 该 skill 可被复制到其他项目中作为单独 skill 使用
- 该 skill 使用一个单一名称进行 `$` 调用
- 验证结果明确记录

## 写回候选

仅当后续反复证明“独立安装 skill 的最小结构要求”是稳定约束时，才考虑写回 `docs/facts/*`。
